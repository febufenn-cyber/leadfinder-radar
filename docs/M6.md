# M6 — secure MCP control server

M6 completes the current LeadFinder roadmap with an owner-only Model Context Protocol surface. It exposes bounded typed tools for inspection and carefully constrained workflow operations without creating an unattended outreach path.

## Tool inventory

| Tool | Behavior | Mutation |
|---|---|---|
| `health` | Process and database health | No |
| `search_leads` | Bounded newest-first search with cursor pagination | No |
| `get_lead` | One lead with bounded post, draft, and send history | No |
| `stats` | Period/pack lead outcomes and LLM cost | No |
| `redraft` | Archive active drafts, generate replacement variants | Yes, reversible |
| `mute` | Add normalized keyword/community mute | Yes, idempotent |
| `request_approval_code` | Deliver a short-lived value to owner Telegram | Yes, challenge only |
| `approve` | Consume one challenge and reuse existing approval/send service | Yes, destructive |

There is no arbitrary SQL, shell, filesystem-write, prompt-edit, lead-state, direct-platform-post, batch-approve, or automatic follow-up tool.

## Architecture

```text
MCP client
  → stdio by default
  → FastMCP typed tool schema
  → timeout + per-process rate limit
  → redacted append-only mcp_tool_call audit
  → existing LeadFinder services and state machines
  → PostgreSQL
```

Network transport is optional. The default is stdio, where the MCP client launches `python -m app.mcp.server` as a child process. Streamable HTTP must be selected explicitly and requires a bearer token. Non-loopback binding additionally requires `MCP_ALLOW_REMOTE=true`; use TLS and network controls outside the process.

## Human approval invariant

Conversational access does not make the model or MCP client the human approver.

1. The client calls `request_approval_code(lead_id, variant)`.
2. LeadFinder validates that the lead is still awaiting approval and binds a challenge to the exact active draft.
3. A six-digit value is sent only to `TELEGRAM_CHAT_ID` through the configured bot.
4. The owner reads it in Telegram and supplies it to `approve`.
5. LeadFinder consumes the single-use challenge and calls the existing copy-mode `approve()` or API-mode `queue_send()` implementation.

The plaintext value is not returned by MCP or stored. The database contains only a random salt, HMAC-SHA256, exact draft SHA-256, expiry, attempt count, and used timestamp. Editing or redrafting invalidates the challenge. Existing API-send jitter, cancellation, caps, cooldowns, quiet hours, halts, and approval-event ordering remain unchanged because M6 reuses those services.

## Configuration

Copy `.env.example` to `.env` and keep the default local mode:

```text
MCP_TRANSPORT=stdio
MCP_BIND_HOST=127.0.0.1
MCP_BIND_PORT=8101
MCP_ALLOW_REMOTE=false
MCP_MAX_CALLS_PER_MINUTE=60
MCP_TOOL_TIMEOUT_SECONDS=30
```

Approval tools additionally require:

```text
TELEGRAM_BOT_TOKEN=...
TELEGRAM_CHAT_ID=...
MCP_APPROVAL_SECRET=...
MCP_APPROVAL_CODE_TTL_SECONDS=300
MCP_APPROVAL_MAX_ATTEMPTS=5
```

Generate the approval secret independently:

```bash
python -c 'import secrets; print(secrets.token_urlsafe(48))'
```

Do not reuse the Telegram token, MCP bearer token, Reddit secret, Threads token, or another platform credential.

### Optional loopback HTTP

```text
MCP_TRANSPORT=streamable-http
MCP_BIND_HOST=127.0.0.1
MCP_BIND_PORT=8101
MCP_AUTH_TOKEN=<long random bearer token>
MCP_ALLOW_REMOTE=false
```

The optional `deploy/leadfinder-mcp.service` runs this loopback HTTP mode. Do not bind publicly without TLS, a firewall/private network, and an explicit security review.

## Local stdio client setup

A secret-free template is in `config/mcp/leadfinder-stdio.json.example`:

```json
{
  "mcpServers": {
    "leadfinder": {
      "command": "uv",
      "args": [
        "--directory",
        "/opt/leadfinder-radar",
        "run",
        "python",
        "-m",
        "app.mcp.server"
      ],
      "env": {"MCP_TRANSPORT": "stdio"}
    }
  }
}
```

The child process reads the normal LeadFinder `.env` from its working directory. Never embed production credentials in a client configuration committed to Git.

## Deployment

### Stdio-only deployment

No daemon is required. Update the repository and migration, then configure the MCP client to launch the server:

```bash
cd /opt/leadfinder-radar
git pull origin main
uv sync --frozen
uv run alembic upgrade head
uv run python scripts/mcp_smoke.py
```

### Optional loopback HTTP service

```bash
sudo cp deploy/leadfinder-mcp.service /etc/systemd/system/leadfinder-mcp.service
sudo systemctl daemon-reload
sudo systemctl enable --now leadfinder-mcp
sudo systemctl status leadfinder-mcp --no-pager
```

Before enabling it, set `MCP_TRANSPORT=streamable-http` and `MCP_AUTH_TOKEN` in `/opt/leadfinder-radar/.env`. The service file forces loopback binding.

## Smoke checks

In-memory protocol and database check:

```bash
uv run python scripts/mcp_smoke.py --in-memory
```

Full stdio subprocess check:

```bash
uv run python scripts/mcp_smoke.py
```

Both checks list the exact eight tools and call `health`. CI runs the in-memory version after migration verification.

## Auditing and error behavior

Every tool call appends an `mcp_tool_call` event containing:

- tool name
- success/failure
- duration
- bounded redacted arguments
- sanitized error category

Keys containing token, secret, password, authorization, auth, or code are replaced by `[redacted]`. Client errors do not include stack traces or database/credential details. Internal failures remain in server logs.

## Redraft behavior

`redraft` is permitted only while a lead is in `drafted` state. Guidance is limited to 500 characters and placed inside the same hardened untrusted payload as post content. It cannot override truthfulness, community rules, language matching, word limits, or human approval.

Before active variants change, all previous drafts are copied into `draft_revisions`. The service rechecks the lead state and exact active draft IDs after generation; concurrent edits or approvals cause the replacement to abort. Only the SHA-256 of guidance is stored in provenance.

## Mute behavior

`mute` accepts only `keyword` or `community`. Values are stripped, lowercased, bounded, and passed through the existing deduplicating mute service. A named offer pack must be enabled.

## Database migrations

- `008` — `draft_revisions`
- `009` — `mcp_approval_challenges`

CI verifies a clean upgrade and an explicit `008 → 009` upgrade. Apply production migrations before using M6:

```bash
uv run alembic upgrade head
```

## Rollback

1. Disable or disconnect the MCP client/service first.
2. M0–M5 worker, bot, dashboard, sending, and watcher services continue independently.
3. Revert the relevant M6 merge commit if code rollback is required.
4. Database downgrade is optional and should be done only after confirming no retained M6 evidence is required:

```bash
uv run alembic downgrade 008
```

Downgrading removes approval challenges. Draft revisions remain at revision `008`; downgrading further removes them.

## Troubleshooting

### Server starts but health fails

Verify `DATABASE_URL`, PostgreSQL availability, and migration head:

```bash
uv run alembic current
uv run alembic upgrade head
```

### Approval challenge is unavailable

Confirm Telegram credentials and `MCP_APPROVAL_SECRET`. The service intentionally refuses console fallback for verification delivery.

### Verification value is rejected

Request a new one. It may have expired, exceeded attempts, been consumed, or been invalidated by a draft change.

### HTTP returns unauthorized

Confirm `MCP_AUTH_TOKEN` and that the client sends it as a bearer token. Authentication is mandatory for streamable HTTP.

### Remote binding is rejected

This is intentional. Keep loopback mode or set `MCP_ALLOW_REMOTE=true` only behind TLS and network controls.

## Definition of done and soak

M6 code completion requires all five slices merged, migration head `009`, green lint/migration/PostgreSQL tests, exact tool inventory, no open M6 PRs, and final `main` confirmation.

The separate roadmap requirement to run unattended for 48 hours cannot be proven by a single interactive build session. Treat the production soak as pending until telemetry demonstrates it.
