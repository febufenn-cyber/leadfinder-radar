# M5 operator guide

M5 turns LeadFinder's accumulated decisions and owner edits into reviewable evidence. It does not
change the human send gate and it does not apply prompt changes automatically.

## Deploy

```bash
git pull origin main
uv sync --frozen
uv run alembic upgrade head
sudo systemctl restart leadfinder-worker leadfinder-bot leadfinder-dashboard
```

Migration `007` creates `review_labels`. The dashboard and worker can start only after the
migration is applied.

## Weekly classifier review

The worker nudges the owner once each Monday when reviewable decisions exist. In the authorized
Telegram chat:

```text
/review10
/review10 robofox_web
```

The queue alternates surfaced and suppressed predictions across packs. Label each card:

- **Demand** — this really is a buying-intent lead.
- **Not lead** — the classifier decision should be negative.
- **Skip** — insufficient context or not worth using as evaluation evidence.

A second tap updates the existing label and appends another audit event; it does not create a
duplicate `review_labels` row.

## Evaluation dashboard

Open `/evals` behind the existing dashboard auth wall. It reports:

- classifier TP, FP, FN, TN, precision, and recall by pack
- worked, replied, conversation, and won outcomes by pack
- chosen A/B/C variant performance
- gold-edit count and average draft-change magnitude
- post-to-alert p50, LLM spend, and cost per surfaced lead

Treat small samples cautiously. Do not tune a pack from one or two labels.

## Monthly prompt proposal

On the first day of each month, each pack with at least three gold edits from the previous 90 days
is analyzed. The result is stored as an event:

```text
kind = prompt_tuning_proposal
payload.applied = false
```

The proposal includes supporting draft ids, repeated edit patterns, conservative rule suggestions,
and overfitting risks. It never writes pack, persona, prompt, or few-shot files.

Manual run:

```bash
uv run python scripts/propose_prompt_tuning.py
```

Before implementing a proposal:

1. Read the full event payload and inspect its referenced gold drafts.
2. Reject rules supported by only one sample.
3. Reject anything that adds invented claims, promotional pressure, or unattended behavior.
4. Make a small prompt/few-shot change in a normal reviewed commit.
5. Compare `/evals` after enough new labels and outcomes accumulate.

## CI

Every pull request and push to `main` runs:

```text
Ruff → Alembic upgrade → full pytest suite against PostgreSQL 16
```

Do not deploy a red `main` branch. If migration verification fails, fix the migration chain rather
than bypassing the step.
